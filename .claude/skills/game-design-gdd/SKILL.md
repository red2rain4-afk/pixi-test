# PixiJS 게임 디자인 문서(GDD) 작성 가이드

이 스킬은 PixiJS 웹 게임을 위한 게임 디자인 문서(GDD)를 작성하고 관리하는 방법을 안내합니다.

## 핵심 원칙

### 1. Living Document 접근
- GDD는 **한 번 작성하고 끝이 아닙니다**
- 게임 개발과 함께 **지속적으로 진화**해야 합니다
- 프로토타입 결과, 플레이테스트 피드백을 즉시 반영하세요

### 2. 간결함 우선
- 100페이지 GDD보다 **5페이지 명확한 GDD**가 낫습니다
- 읽지 않는 문서는 쓸모없습니다
- 시작은 1페이지로, 필요할 때만 확장하세요

### 3. 경험 중심
- "30가지 무기" 같은 **기능 나열**을 피하세요
- 대신 "플레이어는 X를 느낀다" 형식의 **경험 설명**을 우선하세요

## GDD 작성 워크플로우

### Step 1: 상황 파악

사용자에게 다음을 확인하세요:

1. **현재 단계**
   - 새로운 게임 컨셉인가? → 1페이지 GDD부터 시작
   - 프로토타입이 있는가? → 풀 GDD 고려
   - 기존 GDD가 있는가? → 업데이트 모드

2. **게임 유형**
   - 내러티브 중심 → 세계관 문서 별도 필요
   - 메커니즘 중심 → 짧고 빠른 프로토타입 중심
   - 퍼즐/전략 → 메커니즘 상세 설명 중요

3. **목표**
   - 팀 공유용? → 명확한 비전 전달 중심
   - 개인 정리용? → 프로토타입 가이드 중심
   - 투자/퍼블리싱용? → 시장성, USP 강조

### Step 2: 템플릿 선택 및 생성

#### 1페이지 GDD (프로젝트 초기)

**사용 시점:**
- 게임 아이디어를 막 떠올렸을 때
- 프로토타입 전 컨셉 정리
- 빠른 의사소통이 필요할 때

**작성 가이드:**
1. 게임 개요: 30자 이내 한 줄 요약
2. 핵심 경험: 플레이어가 느낄 감정 2-3개
3. 핵심 메커니즘: 플레이어가 반복하는 행동 3개 이내
4. 비주얼: 아트 스타일, 색상 팔레트 (간략히)
5. 기술 스택: PixiJS 버전, 물리엔진 여부
6. MVP 기준: 프로토타입에 필수인 기능 3-5개

**템플릿:** `assets/gdd-template-onepage.md`

#### 풀 GDD (본격 개발 단계)

**사용 시점:**
- 프로토타입이 재미있음을 확인한 후
- 팀원이 합류하여 상세 가이드 필요
- 전체 게임 스코프를 확정할 때

**작성 우선순위:**
1. **필수 (먼저 작성):** 1-3장 (게임 개요, 핵심 경험, 게임플레이 메커니즘)
2. **중요 (다음 작성):** 4-7장 (시스템, 레벨, 비주얼, UI/UX)
3. **선택 (필요 시):** 8-10장 (기술 사양, 마일스톤, 플레이테스트)

**템플릿:** `assets/gdd-template-full.md`

### Step 3: 디자인 방법론 적용

#### Core, Focus & Power 워크시트

GDD 작성 전 이 질문들에 답하세요:

```markdown
## Core (핵심)
**핵심 동사:** [플레이어가 하는 핵심 행동 1-3개]
예: "점프한다", "쏜다", "피한다"

**30초 게임 루프:**
[행동 1] → [결과] → [행동 2] → [결과] → 반복

## Focus (초점)
**핵심 감정:** [플레이어가 느껴야 할 감정]
예: 긴장감, 성취감, 호기심

**전달 메시지:** [게임이 탐구하는 주제]
예: "도전과 극복", "협력의 중요성"

## Power (힘)
**마스터리 대상:** [플레이어가 숙달할 것]
예: "타이밍과 반사신경", "전략적 사고"

**성장 방식:** [레벨업 / 능력 획득 / 스킬 향상]
```

이 워크시트 결과를 GDD의 "핵심 경험" 섹션에 반영하세요.

**상세 가이드:** `references/design-methodologies.md` 참조

#### MDA 프레임워크

**역방향 디자인:**
1. **Aesthetics (목표 감정)** 정의 → "도전과 성취감"
2. **Dynamics (플레이 행동)** 도출 → "리스크/보상 의사결정"
3. **Mechanics (구현 규칙)** 설계 → "체력 시스템, 체크포인트"

이 순서로 GDD를 작성하면 **경험 중심** 디자인이 됩니다.

### Step 4: PixiJS 특화 내용 작성

#### 기술 사양 섹션 체크리스트

```markdown
## 기술 사양 (PixiJS)

### 기본 설정
- PixiJS 버전: [v7.x / v8.x]
- 해상도: [1280x720 / 1920x1080]
- 타겟 FPS: [60]

### 아키텍처
- 물리 엔진: [Matter.js / 없음]
- 상태 관리: [FSM / Redux / 없음]
- 씬 관리: [씬 스택 방식]

### 성능 목표
- 최대 동시 스프라이트: [예: 300개]
- 파티클 제한: [예: 500개]
- 메모리 사용량: [예: < 200MB]
- 로딩 시간: [예: < 3초]

### 최적화 전략
- [ ] 스프라이트 배칭
- [ ] 텍스처 아틀라스
- [ ] 오프스크린 컬링
- [ ] 오브젝트 풀링
```

**중요:** 이 수치들은 프로토타입 단계에서 **실제 측정**하여 작성하세요.

#### PixiJS 코드 예시 포함 (선택)

GDD에 기술적 구현 힌트를 넣고 싶다면:

```javascript
// 예시: 플레이어 이동 (WASD)
const keys = {};
window.addEventListener('keydown', (e) => keys[e.key] = true);
window.addEventListener('keyup', (e) => keys[e.key] = false);

app.ticker.add(() => {
  if (keys['w']) player.y -= 5;
  if (keys['s']) player.y += 5;
  if (keys['a']) player.x -= 5;
  if (keys['d']) player.x += 5;
});
```

이런 코드 스니펫은 프로그래머가 아닌 디자이너와 협업할 때 유용합니다.

### Step 5: GDD 검증

#### 자동 검증 스크립트 실행

```bash
python3 scripts/validate_gdd.py [프로젝트명]-gdd.md
```

**검증 항목:**
- 필수 섹션 존재 확인
- 플레이스홀더 체크 (TODO, [예시] 등)
- 섹션별 내용 충분성
- PixiJS 특화 내용 포함 여부
- 게임 루프 설명 명확성
- 버전 정보 존재

#### 수동 검증 체크리스트

GDD를 다른 사람에게 보여주기 전 확인:

- [ ] **5분 안에 읽을 수 있는가?** (1페이지 GDD 기준)
- [ ] **게임의 핵심 경험이 명확한가?**
- [ ] **플레이 영상 없이도 게임을 상상할 수 있는가?**
- [ ] **기술적으로 실현 가능한가?** (PixiJS 제약 고려)
- [ ] **"재미있을 것 같다"는 느낌이 드는가?**

### Step 6: GDD 유지보수

#### 업데이트 시점

다음 상황에서 **반드시** GDD 업데이트:

1. **프로토타입 후**
   - 의도하지 않은 재미 발견
   - 기술적 제약 확인
   - 예: "점프 메커니즘보다 대시가 더 재미있음" → Core 변경

2. **플레이테스트 후**
   - 플레이어가 이해하지 못한 부분
   - 난이도 조정
   - 예: "튜토리얼이 너무 김" → UI/UX 섹션 수정

3. **마일스톤 완료 후**
   - 새로운 기능 추가
   - 스코프 변경
   - 예: "레벨 10개에서 5개로 축소" → 레벨 디자인 섹션 수정

#### 버전 관리

GDD 하단에 변경 이력 테이블 유지:

```markdown
## 변경 이력

| 날짜 | 버전 | 변경 내용 | 작성자 |
|------|------|-----------|--------|
| 2026-02-03 | 1.0 | 초안 작성 | 김철수 |
| 2026-02-10 | 1.1 | 프로토타입 결과 반영: 대시 메커니즘 추가 | 김철수 |
| 2026-02-20 | 2.0 | 플레이테스트 피드백: 난이도 하향, 레벨 5개로 축소 | 김철수 |
```

## 자주 묻는 질문 (FAQ)

### Q: GDD를 언제 작성해야 하나요?
A: **게임 아이디어가 생긴 직후, 코드 작성 전**에 1페이지 GDD를 작성하세요. 프로토타입이 재미있음을 확인한 후 풀 GDD로 확장하세요.

### Q: 혼자 개발하는데도 GDD가 필요한가요?
A: 네. GDD는 팀 공유용이기도 하지만 **자신의 비전을 명확히 정리**하는 도구입니다. 개발 중 "이게 재미있을까?" 헷갈릴 때 GDD로 돌아가세요.

### Q: GDD가 너무 길어지면?
A: 1) 핵심 내용만 SKILL.md에 남기고 나머지는 별도 파일로 분리 (예: worldbuilding.md, level-design.md), 2) "필수 / 있으면 좋음 / 불필요" 분류 후 불필요한 내용 삭제하세요.

### Q: 프로토타입 결과가 GDD와 다르면?
A: **프로토타입을 따르세요.** 실제 플레이가 기획서보다 중요합니다. GDD를 업데이트하여 발견한 재미를 반영하세요.

### Q: PixiJS 성능 목표를 어떻게 정하나요?
A: 1) 간단한 프로토타입 제작, 2) Chrome DevTools로 FPS, 메모리 측정, 3) 최악의 경우 (적 500개 등) 테스트, 4) 측정값을 기반으로 목표 설정하세요.
