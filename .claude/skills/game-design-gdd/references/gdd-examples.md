# 유명 게임 GDD 사례 분석

이 문서는 실제 게임 개발에서 사용된 GDD 사례와 교훈을 정리합니다.

---

## 1. Grand Theft Auto - "Race'n'Chase" (1995)

**원본 GDD:** 초기 버전은 "Race'n'Chase"라는 제목의 레이싱 게임 컨셉

### 핵심 교훈

#### 1.1 유연한 비전
- **초기 컨셉:** 경찰 vs 도둑 레이싱 게임
- **실제 결과:** 오픈월드 크라임 액션 게임
- **교훈:** GDD는 발견한 재미에 맞춰 진화해야 함

#### 1.2 프로토타입에서 발견
- 플레이테스트 중 "자유롭게 돌아다니는 것"이 더 재미있음을 발견
- 원래 목표(레이싱)보다 부수적 요소(탐험)가 핵심이 됨
- **교훈:** 플레이테스트는 예상 밖의 재미를 찾는 도구

### 1.3 적용 가이드

**당신의 게임에서:**
- [ ] 초기 컨셉에 집착하지 말 것
- [ ] 플레이테스트에서 "플레이어가 가장 많이 하는 행동" 관찰
- [ ] 의도하지 않은 재미를 발견하면 그쪽으로 방향 전환 고려

## 2. Grim Fandango (1998) - LucasArts

**장르:** 어드벤처 게임
**특징:** 상세한 세계관 문서

### 핵심 교훈

#### 2.1 세계관 우선 설계
Grim Fandango의 GDD는 게임 메커니즘보다 **세계관 설명**에 더 많은 지면 할애

**구조:**
1. 세계 설정 (30 페이지)
2. 캐릭터 배경 (20 페이지)
3. 스토리 흐름 (15 페이지)
4. 게임 메커니즘 (5 페이지)

**교훈:** 내러티브 중심 게임은 세계관 문서가 GDD의 핵심

#### 2.2 비주얼 레퍼런스
- 컨셉 아트가 텍스트보다 먼저
- "Film Noir + 멕시칸 Day of the Dead" 무드 보드
- **교훈:** 아트 스타일은 말보다 이미지로

### 2.3 적용 가이드 (내러티브 게임)

**당신의 게임이 스토리 중심이라면:**
- [ ] 세계관 문서 별도 작성 (references/worldbuilding.md)
- [ ] 주요 캐릭터 배경 상세히
- [ ] 비주얼 레퍼런스 이미지 수집 (assets/moodboard/)

**당신의 게임이 메커니즘 중심이라면:**
- [ ] 세계관은 간략히 (1-2 페이지)
- [ ] 메커니즘 설명에 집중

## 3. Doom (1993) - id Software

**장르:** FPS
**특징:** 극도로 짧은 GDD (1페이지)

### 핵심 교훈

#### 3.1 1페이지 GDD의 힘
Doom의 초기 GDD는 **단 1페이지**:

```
제목: Doom
장르: 빠른 속도의 1인칭 슈팅 게임
목표: 괴물을 죽이고 탈출하라
핵심 경험: 압도적인 화력과 속도감
```

**교훈:** 핵심 비전이 명확하면 짧은 GDD로 충분

#### 3.2 기술 우선 접근
- John Carmack이 먼저 3D 엔진 프로토타입 제작
- 디자인은 엔진 기능에 맞춰 진화
- **교훈:** PixiJS 프로젝트에서는 "기술적으로 가능한 것" 먼저 확인

### 3.3 적용 가이드

**PixiJS 프로젝트에서:**
1. **기술 프로토타입 먼저**
   ```javascript
   // 예: 파티클 시스템이 얼마나 성능을 잡아먹는지 먼저 테스트
   const particleTest = new ParticleContainer(1000);
   // 60 FPS 유지되는가? → 게임 디자인에 반영
   ```

2. **제약사항을 디자인으로**
   - "500개 이상 스프라이트는 느림" → 게임은 최대 300개 적으로 제한
   - 제약이 디자인을 결정

## 4. Braid (2008) - Jonathan Blow

**장르:** 퍼즐 플랫포머
**특징:** "메커니즘이 곧 메시지"

### 핵심 교훈

#### 4.1 Theme-First Design (주제 우선 디자인)
- **주제:** 시간, 후회, 관계
- **메커니즘:** 시간 되감기
- **결과:** 메커니즘이 주제를 전달

**프로세스:**
1. 전달하고 싶은 주제 정의
2. 주제를 표현하는 메커니즘 설계
3. 메커니즘을 활용한 퍼즐 제작

#### 4.2 적용 가이드 (테마 중심 게임)

**당신의 게임에 주제가 있다면:**
```markdown
## Theme-to-Mechanics 워크시트

**주제/메시지:**
[예: "협력의 중요성"]

**주제를 표현하는 메커니즘:**
[예: "혼자서는 해결 불가능한 퍼즐, 2명이 협력해야 해결"]

**메커니즘을 활용한 레벨:**
[예: "플레이어 A는 스위치, 플레이어 B는 문 통과"]
```

## 5. Celeste (2018) - Maddy Thorson

**장르:** 정밀 플랫포머
**특징:** 접근성 옵션 풍부

### 핵심 교훈

#### 5.1 코어 vs 메타 분리
Celeste의 디자인 문서는 **두 부분**으로 구성:

**Part 1: Core Mechanics (핵심 메커니즘)**
- 점프, 대시, 벽타기
- 정밀한 컨트롤
- 도전적인 난이도

**Part 2: Accessibility (접근성)**
- 무한 스태미나 옵션
- 슬로우 모션 옵션
- 무적 모드

**교훈:** 핵심 경험과 접근성 옵션은 별개. 둘 다 중요하지만 분리해서 설계

#### 5.2 적용 가이드

**PixiJS 게임에서 접근성:**
```markdown
## 접근성 옵션 체크리스트

- [ ] 난이도 선택 (Easy / Normal / Hard)
- [ ] 색맹 모드 (색상 대비 조정)
- [ ] 컨트롤 커스터마이징
- [ ] 게임 속도 조정 (0.5x ~ 2.0x)
- [ ] 자막/텍스트 크기 조정
```

**구현 예시:**
```javascript
// 게임 속도 조정
const gameSpeed = settings.speed || 1.0; // 0.5 ~ 2.0
app.ticker.speed = gameSpeed;
```

## 6. 인디 게임 GDD 패턴 분석

### 6.1 작은 팀 / 개인 개발자 패턴

**공통점:**
- GDD는 5-10페이지 이하
- Living Document로 관리 (Notion, Google Docs)
- 이미지 > 텍스트

**예시 구조:**
```
1. 핵심 컨셉 (1페이지)
2. 레퍼런스 게임 (이미지 위주)
3. 핵심 메커니즘 (2-3페이지)
4. 아트 스타일 (무드보드)
5. 마일스톤 (간트 차트)
```

### 6.2 PixiJS 인디 게임 추천 구조

**1페이지 GDD → 프로토타입 → 풀 GDD 순서:**

#### Week 1: 1페이지 GDD
```markdown
# [게임 제목]
- 장르: [2D 플랫포머]
- 핵심: [점프와 대시로 장애물 피하기]
- 레퍼런스: [Super Meat Boy]
- MVP: [5개 레벨]
```

#### Week 2-3: 프로토타입 + 플레이테스트
- 그레이박스 구현
- 컨트롤 필 테스트

#### Week 4+: 풀 GDD 작성
- 프로토타입 결과 반영
- 전체 레벨 구조 확정

## 7. 안티패턴: 실패하는 GDD

### 7.1 100페이지 GDD (읽지 않는 문서)
**문제:**
- 작성하는 데 한 달 소요
- 아무도 읽지 않음
- 게임 변경 시 업데이트 안 됨

**해결:**
- 5페이지로 줄여라
- 나머지는 별도 문서로 (references/)

### 7.2 Feature List GDD (기능 나열)
**문제:**
- "30가지 무기, 50개 레벨, 100개 적" 같은 숫자 나열
- "왜 재미있는지" 설명 없음

**해결:**
- 기능보다 **경험** 중심으로 작성
- "플레이어는 X를 느낀다" 형식

### 7.3 고정된 GDD (변경 불가)
**문제:**
- "GDD에 적혀있으니 이대로 해야 함"
- 플레이테스트 피드백 무시

**해결:**
- GDD는 Living Document
- 매 스프린트마다 업데이트

## 8. PixiJS 게임 GDD 템플릿 추천

### 8.1 선택 가이드

| 상황 | 추천 템플릿 |
|------|-------------|
| 막 시작, 컨셉 정리 중 | `gdd-template-onepage.md` |
| 프로토타입 완료, 본격 개발 | `gdd-template-full.md` |
| 내러티브 중심 게임 | Full GDD + `references/worldbuilding.md` |
| 메커니즘 중심 게임 | 1-page GDD + 빠른 프로토타입 |

### 8.2 PixiJS 특화 추가 섹션

**Full GDD에 추가 권장:**

```markdown
## PixiJS 기술 상세

### 씬 관리
- [씬 전환 방식: Fade / Slide / Instant]
- [씬 구조: MainMenu / Game / GameOver]

### 스프라이트 관리
- [최대 동시 스프라이트: 300개]
- [스프라이트 풀링 사용: Yes/No]

### 에셋 로딩
- [총 에셋 크기: ~10MB]
- [로딩 전략: Preload all / Lazy load]

### 성능 최적화
- [타겟 FPS: 60]
- [최악 케이스 테스트: 500개 파티클]
```
