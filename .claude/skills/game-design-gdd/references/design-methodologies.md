# 게임 디자인 방법론 가이드

이 문서는 게임 디자인 시 적용할 수 있는 검증된 방법론들을 설명합니다.

---

## 1. Core, Focus & Power 방법론

**출처:** Gamasutra / Game Developer
**목적:** 명확하고 구체적인 의도를 가진 게임 설계

### 핵심 개념

#### Core (핵심)
게임의 **본질적인 경험**. 플레이어가 게임에서 하는 가장 중요한 것.

**질문:**
- "플레이어는 이 게임에서 무엇을 하는가?"
- "이 게임의 핵심 동사는 무엇인가?" (예: 점프하다, 쏘다, 퍼즐을 풀다)

**예시:**
- **슈퍼 마리오:** 점프하고 플랫폼을 이동한다
- **테트리스:** 블록을 회전하고 배치한다
- **포탈:** 포탈을 배치하고 공간을 이동한다

**적용 방법:**
1. 게임의 핵심 동사 1-3개 정의
2. 이 동사들이 30초마다 반복되는지 확인
3. 핵심이 아닌 요소는 최소화

#### Focus (초점)
게임이 플레이어에게 **전달하고자 하는 메시지나 감정**.

**질문:**
- "플레이어가 이 게임에서 느껴야 할 감정은?"
- "이 게임이 탐구하는 주제는?"

**예시:**
- **다크 소울:** 도전과 극복 → 긴장감과 성취감
- **저니:** 연결과 발견 → 경이로움과 외로움
- **언더테일:** 선택의 무게 → 도덕적 고민

**적용 방법:**
1. 핵심 감정 1-2개 선택
2. 모든 디자인 결정이 이 감정을 강화하는지 질문
3. 초점에 맞지 않는 요소 제거

#### Power (힘)
플레이어가 게임을 통해 얻는 **역량과 성장감**.

**질문:**
- "플레이어는 무엇을 마스터하는가?"
- "플레이어는 어떻게 성장하는가?"

**예시:**
- **격투 게임:** 콤보와 타이밍 마스터리 → 스킬 파워
- **RPG:** 레벨업과 장비 → 수치적 파워
- **메트로이드배니아:** 능력 획득 → 탐험 파워

**적용 방법:**
1. 명확한 성장 곡선 설계
2. 초반 약함 → 후반 강함의 차이를 체감하게
3. 플레이어 스킬 vs 캐릭터 스펙의 균형 설정

### 워크시트: Core, Focus & Power

게임 디자인 시 다음 질문에 답하세요:

```markdown
## Core (핵심)
**핵심 동사:** [예: 점프하다, 쏘다, 피하다]
**핵심 루프 (30초):** [행동 → 결과 → 반복]
**핵심이 아닌 것:** [제거할 요소]

## Focus (초점)
**핵심 감정:** [예: 긴장감, 성취감, 호기심]
**전달 메시지:** [게임이 탐구하는 주제]
**감정 강화 방법:** [비주얼, 사운드, 메커니즘]

## Power (힘)
**마스터리 대상:** [플레이어가 숙달하는 스킬]
**성장 방식:** [레벨업 / 능력 획득 / 스킬 향상]
**초반 vs 후반:** [차이 설명]
```

## 2. Rational Game Design (합리적 게임 디자인)

**목적:** 데이터와 논리를 기반으로 게임 시스템을 설계

### 핵심 원칙

#### 2.1 플레이어 목표와 시스템 정렬
게임 시스템은 플레이어의 목표를 명확히 지원해야 함.

**나쁜 예:**
- 게임 목표: "적 제거"
- 실제 보상: "숨어서 적 피하기가 더 유리함"
- 문제: 시스템이 목표를 방해함

**좋은 예:**
- 게임 목표: "적 제거"
- 보상: "적 제거 시 경험치, 자원 획득"
- 결과: 시스템이 목표를 강화함

#### 2.2 피드백 루프 설계
플레이어의 행동 → 게임의 반응 → 플레이어의 이해

**필수 요소:**
- **즉각성:** 행동 후 0.1초 이내 반응
- **명확성:** 무슨 일이 일어났는지 확실히 알 수 있음
- **일관성:** 같은 행동은 항상 같은 결과

**PixiJS 구현 예:**
```javascript
// 좋은 피드백: 즉각적, 명확, 일관적
player.on('hit', () => {
  player.tint = 0xff0000; // 빨간색으로 깜빡 (시각적 피드백)
  sfx.play('hurt'); // 사운드 피드백
  health -= 10; // 수치 변화
  setTimeout(() => player.tint = 0xffffff, 100); // 원래 색으로
});
```

#### 2.3 의사결정 깊이 (Decision Depth)
플레이어가 의미 있는 선택을 할 수 있어야 함.

**나쁜 선택:**
- "A를 선택하면 항상 유리함" → 실제로는 선택이 아님
- "랜덤으로 결정됨" → 전략이 없음

**좋은 선택:**
- "상황에 따라 A와 B가 각각 유리함"
- "단기 이득 vs 장기 이득" 트레이드오프

## 3. MDA 프레임워크 (Mechanics, Dynamics, Aesthetics)

**출처:** Robin Hunicke, Marc LeBlanc, Robert Zubek
**목적:** 게임을 체계적으로 분석하고 설계

### 3.1 Mechanics (메커니즘)
게임의 **규칙과 시스템**. 코드로 구현되는 부분.

**예시:**
- "점프 버튼을 누르면 플레이어는 10 유닛 위로 이동"
- "적과 충돌하면 체력 1 감소"
- "코인 100개 모으면 생명 1 증가"

### 3.2 Dynamics (역학)
메커니즘이 **플레이 중에 상호작용하며 생성되는 행동**.

**예시 (슈퍼 마리오):**
- **메커니즘:** 점프, 적과 충돌 시 데미지, 위에서 밟으면 적 제거
- **다이나믹스:** "플레이어는 적을 피할지, 제거할지 선택한다" → 리스크/보상 판단

### 3.3 Aesthetics (미학)
플레이어가 경험하는 **감정적 반응**.

**8가지 미학 (MDA 논문):**
1. **Sensation (감각):** 감각적 즐거움 (예: 뷰티풀 비주얼)
2. **Fantasy (판타지):** 상상의 세계 (예: RPG)
3. **Narrative (서사):** 스토리 (예: 어드벤처 게임)
4. **Challenge (도전):** 장애물 극복 (예: 플랫포머)
5. **Fellowship (동료애):** 사회적 즐거움 (예: MMO)
6. **Discovery (발견):** 탐험 (예: 메트로이드배니아)
7. **Expression (표현):** 창의성 (예: 마인크래프트)
8. **Submission (몰입):** 시간 때우기 (예: 캐주얼 게임)

### MDA 역방향 디자인 프로세스

**디자이너 관점:** Mechanics → Dynamics → Aesthetics
**플레이어 관점:** Aesthetics → Dynamics → Mechanics

**적용 방법:**
1. **목표 Aesthetics 정의:** "플레이어가 도전과 성취감을 느끼게"
2. **필요한 Dynamics 도출:** "리스크/보상 의사결정이 필요"
3. **구현할 Mechanics 설계:** "체력 시스템, 적 패턴, 체크포인트"

## 4. Iterative Design (반복적 디자인)

**핵심 원칙:** 완벽한 첫 버전은 없다. 빠르게 만들고, 테스트하고, 개선하라.

### 반복 사이클

```
1. 프로토타입 → 2. 플레이테스트 → 3. 분석 → 4. 개선 → 반복
```

### 4.1 프로토타입 단계별 접근

#### 페이퍼 프로토타입 (Day 1)
- **도구:** 종이, 펜
- **목적:** 핵심 메커니즘 검증
- **질문:** "이 규칙이 재미있는가?"

#### 디지털 그레이박스 (Day 2-3)
- **도구:** PixiJS + 사각형/원 그래픽
- **목적:** 게임 필 검증
- **질문:** "컨트롤이 기분 좋은가?"

#### 수직적 슬라이스 (Week 1-2)
- **도구:** 완성 그래픽 + 사운드
- **목적:** 최종 게임 경험의 작은 버전
- **질문:** "이게 최종 게임 느낌이 맞는가?"

### 4.2 플레이테스트 체크리스트

**관찰할 것:**
- [ ] 플레이어가 목표를 이해하는가?
- [ ] 컨트롤을 배우는 데 얼마나 걸리는가?
- [ ] 어느 부분에서 좌절하는가?
- [ ] 어느 부분에서 웃거나 흥분하는가?
- [ ] 중간에 그만두고 싶어 하는가?

**질문할 것:**
- "무엇을 해야 할지 알았나요?"
- "가장 재미있었던 부분은?"
- "가장 답답했던 부분은?"
- "다시 플레이하고 싶나요?"

**하지 말아야 할 것:**
- 플레이 중 설명하기 (플레이어 혼자 이해해야 함)
- "재미있었나요?" 같은 유도 질문
- 피드백 즉시 변명하기

## 5. PixiJS 특화 디자인 고려사항

### 5.1 웹 게임 제약사항

#### 성능 제약
- **제한:** 브라우저 환경, 다양한 디바이스 성능
- **전략:**
  - 동시 스프라이트 수 제한 (< 500개)
  - 파티클 효과 최적화
  - 텍스처 아틀라스 사용

#### 입력 제약
- **제한:** 키보드/마우스 위주 (모바일 별도 고려 필요)
- **전략:**
  - 마우스 전용 or 키보드 전용 선택
  - 복잡한 키 조합 피하기

#### 세션 길이
- **제한:** 웹 게임은 짧은 세션 선호
- **전략:**
  - 5-10분 세션 설계
  - 빠른 재시작

### 5.2 웹 게임 강점 활용

#### 즉각적인 접근성
- **강점:** 설치 불필요, URL만으로 공유
- **활용:**
  - 튜토리얼 최소화 (바로 플레이 가능하게)
  - 소셜 공유 기능

#### 빠른 업데이트
- **강점:** 배포가 즉각적
- **활용:**
  - A/B 테스트 용이
  - 이벤트/시즌 콘텐츠

## 6. 디자인 실수 피하기 (Common Pitfalls)

### 6.1 Scope Creep (범위 확장)
**문제:** "이 기능도 넣으면 좋겠다" → 끝없는 기능 추가
**해결:**
- MVP (최소 기능 제품) 명확히 정의
- "필수 / 있으면 좋음 / 불필요" 분류
- 완성 후 추가 고려

### 6.2 Tutorial Hell (튜토리얼 지옥)
**문제:** 게임 시작 전 10분 튜토리얼
**해결:**
- "보여주지 말고, 하게 하라"
- 안전한 환경에서 자연스럽게 학습
- 텍스트 최소화, 행동으로 가르치기

### 6.3 False Difficulty (허위 난이도)
**문제:** 어려운 이유가 불명확하거나 불공정함
**해결:**
- 플레이어 실수 vs 게임 문제 구분
- 실패 원인을 명확히 피드백
- "어렵지만 공정함"을 목표로

### 6.4 Feature Bloat (기능 과다)
**문제:** 너무 많은 시스템으로 복잡도 증가
**해결:**
- "더하기"보다 "빼기"
- 각 시스템이 핵심 경험을 강화하는지 질문
- 서로 상호작용하지 않는 시스템 제거
